Code documentation¶

Backend¶

The contract a backend must respect:

class ragger.interface.BackendInterface(host: str, port: int, raises: bool = False)¶
__init__(host: str, port: int, raises: bool = False)¶

Initializes the Backend

Parameters

  • host (str) – The host where to reach the backend.

  • port (int) – The port where to reach the backend

  • raises (bool) – Weither the instance should raises on non-0x9000 response statuses, or not.

abstract both_click() → None¶

Performs a click on both buttons of the device.

This method may be left void on backends connecting to physical devices, where a physical interaction must be performed instead. This will prevent the instrumentation to fail (the void method won’t raise NotImplementedError), but the instrumentation flow will probably get stuck (on further call to receive for instance) until the expected action is performed on the device.

Returns

Nothing

Return type

None

exchange(cla: int, ins: int, p1: int = 0, p2: int = 0, data: bytes = b'') → ragger.interface.RAPDU¶

Formats and sends an APDU to the backend, then receives its response.

The length is automaticaly added to the APDU message.

Parameters

  • cla (int) – The application ID

  • ins (int) – The command ID

  • p1 (int) – First instruction parameter, defaults to 0

  • p1 – Second instruction parameter, defaults to 0

  • data (bytes) – Command data

Raises

ApduException – If the raises attribute is True, this method will raise if the backend returns a status code different from 0x9000

Returns

The APDU response

Return type

RAPDU

Responses¶

Most APDU response are instances of class RAPDU:

class ragger.interface.RAPDU(status: int, data: bytes)¶

However, if the backend has been instantiated with raises=True, APDU responses with a status different from 0x9000 will raises an ApduException:

class ragger.ApduException(sw: int, data: bytes)¶
